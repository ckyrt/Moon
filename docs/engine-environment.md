# Environment Module

## 定位

自然环境系统用于为 Moon Engine 提供接近 3A 游戏的大世界环境表现，包括天空、大气、昼夜循环、天气、风场、植被动态和环境光照联动。

该系统的目标不是做一个单点特效，而是提供一套可扩展、可配置、可裁剪的环境框架，让不同项目可以按需启用，并保持与其他游戏项目隔离。

## 核心目标

- 支持真实、连续、沉浸式的动态自然环境表现
- 支持大场景下的稳定运行与可扩展渲染
- 支持项目级开关，不默认侵入现有游戏逻辑
- 支持编辑器配置、运行时驱动、数据资源解耦
- 支持从基础版到高质量版逐步升级，不强制一次性实现全部效果

## 产品目标描述

Moon Engine 需要实现一个完整的动态自然环境系统，使游戏世界具备类似 GTA、PUBG 一类 3A 游戏中的自然环境表现。系统需要支持物理基础天空与大气渲染、昼夜循环、动态天气、风场驱动、地形与植被联动，以及由环境变化引起的光照和氛围变化。最终目标是在大规模场景中提供自然、连续、真实且可扩展的环境表现，同时保证引擎层面的模块独立性，不影响未启用该系统的其他项目。

## 设计原则

### 1. 模块独立

环境系统应作为独立模块存在，不把逻辑散落到 terrain、render、scene、gameplay 的各个角落。建议采用统一入口，例如：

- `engine/environment/EnvironmentSystem`
- `engine/environment/EnvironmentComponent`
- `engine/environment/EnvironmentProfile`

### 2. 项目隔离

该系统必须默认“按项目启用”，而不是全局强制生效：

- 未启用环境系统的项目，渲染流程和场景逻辑保持原样
- 环境系统使用独立配置、独立资源、独立运行时状态
- 不将天气、时间、风场等状态写死进全局单例
- 通过场景组件、世界配置或项目设置显式启用

### 3. 数据驱动

环境表现尽量通过资源配置驱动，而不是硬编码：

- 时间曲线
- 天气状态机
- 云层参数
- 雾参数
- 风场参数
- 植被响应参数
- 环境光照预设

### 4. 分层实现

先完成“可运行、可联动、可扩展”的基础框架，再逐步提升画质：

- 第一阶段先做时间、天空颜色、主光方向、基础天气切换
- 第二阶段引入体积云、雾、风场和植被动画
- 第三阶段加入更高质量的大气散射、云影、湿地表反馈、远景优化

## 八个子系统

### 1. 天空与大气系统

职责：

- 太阳、月亮、天空穹顶渲染
- 大气散射
- 天空颜色随时间变化
- 地平线、天顶色、太阳盘和环境雾联动

建议能力：

- 基础版：渐变天空 + 太阳方向光
- 进阶版：物理基础 Rayleigh/Mie 散射
- 高质量版：多重散射近似、时间段色彩过渡、远景雾融合

### 2. 昼夜与时间系统

职责：

- 世界时间推进
- 日期、时间倍率、经纬度或简化太阳轨迹
- 驱动太阳方向、月亮方向、天空颜色和环境光变化

建议能力：

- 可暂停、可加速、可固定时刻
- 支持编辑器预览
- 支持脚本和玩法系统读取当前环境时间

### 3. 体积云系统

职责：

- 云层覆盖
- 云体形状和密度变化
- 云层移动
- 云影投射到地表

建议能力：

- 基础版：2D 云图/层云
- 进阶版：屏幕空间或体积云
- 高质量版：多层云体、天气联动、时间段散射变化

### 4. 天气系统

职责：

- 晴天、阴天、雨天、雾天等状态管理
- 天气状态切换和过渡
- 驱动天空、云层、光照、雾、地表湿度、粒子效果

建议能力：

- 支持全局天气
- 预留区域天气能力
- 支持天气权重混合和时间平滑过渡

### 5. 风场系统

职责：

- 提供统一风向、风强、阵风、噪声扰动
- 驱动草地、树木、布料、粒子等自然反馈
- 为后续烟雾、降雨方向、云层移动提供统一输入

建议能力：

- 全局风
- 区域风体积
- 噪声风场贴图或程序风场

### 6. 植被系统

职责：

- 草、灌木、树木的大规模分布与渲染
- LOD、实例化、裁剪
- 风场驱动下的摆动动画

建议能力：

- 与 terrain 解耦，但可读取地形高度、坡度、材质层
- 支持程序分布和手工刷植被
- 支持近景精细动画与远景简化动画

### 7. 地形与自然表层系统

职责：

- 大规模地形渲染
- 地表材质混合
- 与天气和湿度联动的自然表层反馈

建议能力：

- 高度图/瓦片地形
- 地表材质权重混合
- 雨天湿润度、积水感、雾天可见度变化

说明：

地形系统本身仍可保留在 `terrain` 模块内，但环境系统需要定义与其交互的标准接口，而不是直接耦合内部实现。

### 8. 环境光照与氛围系统

职责：

- 主方向光、天空光、环境反射联动
- 雾、曝光、色调和后处理氛围变化
- 天气与时间对场景整体氛围的统一控制

建议能力：

- 环境光颜色和强度曲线
- 雾颜色、密度、高度控制
- 不同天气下的曝光和色调预设

## 推荐架构

### 模块边界

建议新增独立模块：

- `engine/environment`

对外只暴露有限接口：

- `IEnvironmentSystem`
- `EnvironmentState`
- `EnvironmentProfile`
- `WeatherProfile`
- `WindFieldState`
- `TimeOfDayState`

### 与其他模块的关系

- `render`：负责具体渲染通道、shader、GPU 资源
- `terrain`：提供地形高度、材质层、植被分布输入
- `scene`：挂载环境组件和场景级配置
- `core`：提供事件、资源、序列化、时间更新基础设施
- `editor`：提供环境配置面板与实时预览

### 推荐集成方式

环境系统不直接控制所有模块，而是输出统一状态：

- 当前太阳/月亮方向
- 当前天气权重
- 当前风场参数
- 当前雾参数
- 当前环境光参数

其他系统按需读取这些状态进行表现更新。

## 隔离策略

为了“主要独立，项目隔离，不影响其他项目”，建议明确以下规则：

### 启用方式

- 仅当项目或场景显式添加 Environment 功能时才初始化
- 没有 Environment 组件时，不创建相关运行时资源
- 无环境资源时，回退到当前默认天空/光照流程

### 资源隔离

- 环境配置文件单独存放
- 天气、云层、风场、时间曲线使用独立资源类型
- 不复用其他项目的环境默认值，避免隐式串联

### 运行时隔离

- 环境状态与项目 World/Scene 实例绑定
- 不使用跨项目共享的静态环境状态
- 编辑器预览状态和游戏运行状态分离

### 渲染隔离

- 为天空、云、雾、植被风动画增加独立渲染 feature flag
- 默认关闭高级效果
- 不支持的平台自动降级，不阻塞其他项目运行

## 分阶段实现建议

### Phase 1: 可用基础版

- 时间系统
- 太阳方向光
- 基础天空渐变
- 晴天/阴天/雾天切换
- 基础全局雾
- 环境配置资源

目标：

先建立完整控制链路，让世界能随着时间和天气发生统一变化。

### Phase 2: 沉浸增强版

- 体积云或高质量云层
- 风场系统
- 草地/树木风摆
- 雨天参数联动
- 天气平滑过渡
- 环境光照曲线

目标：

让环境从“会变化”提升到“看起来是活的”。

### Phase 3: 高质量大世界版

- 云影
- 区域天气
- 湿地表反馈
- 远景大气透视
- 大规模植被实例化优化
- 更高质量的天空散射模型

目标：

在大场景下接近 3A 开放世界的自然环境表现。

## 性能要求

- 必须支持质量分级和 feature flag
- 必须支持远近景差异化渲染
- 植被必须使用实例化和 LOD
- 云、雾、风动画必须可按平台降级
- 环境更新逻辑应与渲染解耦，避免每帧重复重算全部状态

## MVP 建议

如果以最小可交付版本为目标，优先顺序建议如下：

1. 时间系统
2. 太阳和天空颜色变化
3. 天气状态切换
4. 全局雾
5. 风场参数
6. 草地摆动
7. 云层表现
8. 更高质量大气散射

## 验收标准

- 同一场景可以从清晨平滑过渡到正午、黄昏和夜晚
- 天气切换时，天空、光照、雾和氛围同步变化
- 草地和树叶能够对风产生明显但自然的动态响应
- 在大场景下保持可接受帧率，并可按质量档位降级
- 未启用环境系统的项目不受功能和性能影响

## 总结

自然环境系统应被视为 Moon Engine 的一个独立引擎能力层，而不是若干零散视觉效果的拼接。其关键成功标准有两个：

- 视觉上足够真实和连续
- 工程上足够独立、可裁剪、不会污染其他项目

如果按照上述边界推进，Moon 可以先用较低风险建立统一环境框架，再逐步把画质提升到接近 3A 大世界的表现水准。
